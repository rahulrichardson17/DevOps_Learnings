4. What happens internally when you run kubectl apply -f deployment.yaml

    When we run kubectl apply -f deployment.yaml

    kubectl reads the YAML manifest, converts it into an API request, and sends it to the Kubernetes API Server.
    The API Server authenticates the user, authorizes the request using RBAC, validates the object, and stores the desired state in etcd.
    The Deployment Controller running inside kube-controller-manager watches for the new Deployment. It notices that the desired state exists 
    but the corresponding ReplicaSet does not, so it creates a ReplicaSet and updates the cluster state through the API Server, which is then persisted in etcd.

    Next, the ReplicaSet Controller notices that the desired number of Pods is not running. 
    If the Deployment specifies 3 replicas, it creates 3 Pod objects and updates the API Server, which stores them in etcd.

    The Scheduler watches for Pods that do not have a node assigned. It evaluates the available worker nodes based on CPU, memory, taints, 
    tolerations, affinity rules, and node health, then selects the most suitable node and assigns the Pods to it.

    The kubelet running on the selected worker node notices that Pods have been assigned to its node. 
    It instructs the container runtime, such as containerd, to pull the required image from the container registry if it is not already present locally and then start the containers.

    Once the containers are running, kubelet continuously reports the Pod status back to the API Server, which persists the updated state in etcd.


    **** Every Components update the API Server, and the API Server persists the changes into etcd.

    4a) Who creates the ReplicaSet when a Deployment is created?"
        Deployment controller inside the control manager creates the replica set.

    4b) Who creates Pods from a ReplicaSet?
        ReplicaSet Controller inside the control manager creates the pods.


5. How does kube-scheduler decide where to place a pod?

    kube-scheduler watches for Pods that do not yet have a node assigned. It first filters out nodes that cannot run the Pod based on available CPU, memory, 
    node readiness, taints/tolerations, node selectors, and affinity rules. It then scores the remaining nodes according to scheduling policies 
    such as resource utilization and workload distribution. 
    The highest-scoring node is selected, and the scheduler updates the Pod's nodeName via the API Server. The kubelet on that node then creates and runs the Pod.

7. Explain why kube-apiserver is stateless but etcd is stateful

    kube-apiserver is stateless because it does not persist any cluster data locally. 
    It only acts as a frontend that handles client requests, performs authentication, authorization, validation, 
    and then stores or retrieves data from etcd. Since it does not hold any persistent state, multiple API server instances
    can run behind a load balancer and serve requests interchangeably.

    etcd, on the other hand, is stateful because it is the source of truth for the Kubernetes cluster. 
    It stores all persistent cluster data such as Pods, Deployments, Services, and ConfigMaps, and ensures consistency using the Raft consensus algorithm. 
    If etcd is lost, the cluster state is lost, which is why it requires high availability and backup.

    Pod-to-Pod communication in Kubernetes is handled by the CNI plugin using Linux networking constructs like veth pairs and bridges. 
    For same-node communication, traffic flows locally through a Linux bridge. For cross-node communication, 
    the packet is routed through the host network using either overlay mechanisms like VXLAN (Flannel) or routing-based approaches like BGP (Calico), before reaching the destination node where it is delivered to the target Pod.

    Cross-node Pod communication  
        Pod → veth → bridge → NIC → network → NIC → bridge → veth → Pod (Network Interface Card)

9. What happens when a node goes NotReady?

    When a node becomes NotReady, it means the kubelet has stopped reporting its status or the node lease has expired.
    The node controller in the control plane detects this and marks the node as NotReady in the API server.
    The scheduler stops placing new pods on that node. After a grace period, a NoExecute taint is applied, and existing pods are evicted.
    If those pods are managed by a controller like a Deployment, they are recreated on healthy nodes. This ensures workload continuity despite node failure.

12. How does Horizontal Pod Autoscaler (HPA) work internally?

    Horizontal Pod Autoscaler works as a control loop inside the Kubernetes controller manager. 
    It periodically fetches metrics such as CPU or memory usage from the metrics server or custom metrics APIs. 
    It then compares the observed usage with the target utilization defined in the HPA configuration and calculates the desired number of replicas using a proportional formula. 
    This desired replica count is bounded by minReplicas and maxReplicas and then updated in the Deployment or ReplicaSet. The ReplicaSet controller and scheduler then ensure that 
    the required number of Pods are created and scheduled on available nodes. HPA continuously repeats this process every 15–30 seconds to adjust scaling based on real-time load.

    HPA fetches metrics : kubectl get --raw /apis/metrics.k8s.io
    Calculates the average metrics  Average CPU = (80 + 85 + 90) / 3 = 85%
    Apply HPA formula:
        desiredReplicas = currentReplicas × (currentMetric / targetMetric)


    How often does HPA run?
        Every 15–30 seconds (default)
        Controlled by controller-manager sync loop

    1. Metrics-server collects CPU usage
        ↓
    2. HPA reads metrics
            ↓
    3. HPA calculates desired replicas
            ↓
    4. HPA updates Deployment (replicas field via API Server)
            ↓
    5. Deployment controller updates ReplicaSet
            ↓
    6. ReplicaSet creates/deletes Pods
            ↓
    7. Scheduler assigns nodes
            ↓
    8. kubelet starts containers

13. How do you handle burst traffic (like network traffic spikes in telecom)?

        Burst traffic in Kubernetes is handled using a multi-layer approach. At the edge layer, load balancers and API gateways absorb and 
        distribute sudden spikes. At the application layer, Horizontal Pod Autoscaler scales pods based on metrics like CPU, requests 
        per second, or queue length. At the infrastructure layer, Cluster Autoscaler adds new nodes when pods cannot be scheduled. Additionally, 
        production systems often use message queues like Kafka to buffer traffic and decouple request ingestion from processing. Because Kubernetes scaling 
        is not instantaneous, systems also maintain a buffer of pre-warmed pods and apply rate limiting to handle sudden traffic surges effectively.

15. How do node affinity and pod affinity differ?

    Both node affinity and pod affinity are about where pods should be scheduled
    But they differ in what they match against.

    NA- Place this pod on a node with certain labels
    PA- Place this pod close to other specific pods

    Node affinity selects which node a pod should run on, while pod affinity 
    controls where a pod should run relative to other pods.

20. Node is ready but pods are not scheduling—what could be reasons?

    If a node is in Ready state but pods are not being scheduled,
    it usually indicates scheduling constraints rather than node health issues. Common reasons 
    include insufficient CPU or memory resources, node taints without matching 
    tolerations, node selector or affinity rules not matching, persistent volume binding issues, 
    resource quota limits, or pod anti-affinity constraints. The scheduler evaluates all these conditions before placing a pod. 
    Even if the node is healthy, it will be skipped if it does not satisfy all scheduling requirements.

    kubernetes restarts the containers.

    readiness probe, checks if the application, is ready to recive the traffic
    if it fails, pod stays running and but service stops sending traffic to it.


Rolling Update is the default deployment strategy in Kubernetes. During an update, new Pods are created gradually while old Pods are terminated, ensuring zero or minimal downtime. The behavior is controlled by maxSurge and maxUnavailable.

MaxSurge- How many extra Pods can Kubernetes temporarily create during an update?
maxUnavailable - How many Pods are allowed to be down during the update?

28. Can one Pod communicate with another Pod without a Service?
    Yes. Every Pod gets its own routable IP address in the cluster. Pods can communicate directly using Pod IPs and application ports.
    However, in production we usually use a Service because Pod IPs change when Pods are recreated.

Q: Is containerPort required for a Service to work?
    No. containerPort is optional metadata in the Pod spec. The Service uses targetPort to decide where to send traffic. 
    As long as the application is listening on the targetPort, the Service works even if containerPort is omitted or incorrect.

    containerPort = "The application listens here" (documentation/metadata)
    targetPort    = "The Service sends traffic here" (actual routing)

    In practice, most teams keep them the same:

    containerPort: 80
    targetPort: 80

    because it's clear and avoids confusion, but Kubernetes does not require them to match.

1. Probes allow Kubernetes to check the application state.

    What problem do probes solve?

        Without probes, Kubernetes only knows whether the container process exists. 
        A Flask app can be running but still be:
            Starting up and not ready yet.
            Stuck in a deadlock or infinite loop.
            Connected to a broken dependency (DB/cache) and returning errors.
            Recoverable only by restarting the container.


        Startup Probe   -> Give me time to boot
        Readiness Probe -> Can I take traffic?
        Liveness Probe  -> Should I be restarted?


Q: If the CI/CD runner is new every time, won't all Kubernetes resources be recreated?

    No. Kubernetes stores the cluster state in etcd, not on the CI/CD runner. 
    When kubectl apply runs, Kubernetes compares the desired state in the manifests with the current state 
    in the cluster. Unchanged resources remain unchanged, modified resources are updated, and only missing 
    resources are created. The runner is ephemeral, but the cluster state persists.

Q: What happens when you create an EKS cluster?

    Creating an EKS cluster provisions the managed Kubernetes control plane, including the API server, scheduler, 
    controller manager, and etcd. No worker nodes are created by default. To run workloads, you must add worker nodes 
    through managed node groups, self-managed EC2 instances, or serverless options like AWS Fargate.

    You don't see:
        API Server,Scheduler,etcd,Controller Manager
        as separate resources because AWS manages them for you.


    1. Cluster IAM Role
        This role is assumed by the EKS Control Plane. Sometimes the control plane needs to talk to AWS services.
        Those permissions come from: Cluster IAM Role

    2. Node IAM Role
        Attached to worker nodes (EC2 instances).
        This role is attached to the EC2 worker nodes.
        Why do nodes need permissions?
            Because the kubelet and containers running on the node need AWS access.

            AmazonEKSWorkerNodePolicy
            AmazonEC2ContainerRegistryReadOnly
            AmazonEKS_CNI_Policy

    Node = EC2 Instance (in EKS), In EKS, a worker node is usually an EC2 instance.
    Node Group = Group of EC2 Instances
    A Node Group is a collection of EC2 instances with similar configuration:
    same Instance type, AMI, IAM Role.


Q: Can a Kubernetes administrator log in to the control plane?

    In a self-managed Kubernetes cluster (kubeadm), yes. The administrator can SSH into the control plane node and inspect components such as the API server, scheduler, controller manager, and etcd.
    In managed Kubernetes services such as Amazon Web Services EKS, no. The control plane is managed by the provider and is not directly accessible. 
    Administrators interact with it through Kubernetes APIs, kubectl, and provider-managed logs and monitoring.
