1)Why is OIDC more secure than storing cloud credentials in GitHub Secrets, and how would you implement it?
    OIDC (OpenID Connect) is more secure because it eliminates long-lived cloud credentials from GitHub entirely. 
    Instead of storing a cloud access key in GitHub Secrets, GitHub requests a short-lived identity token at 
    workflow runtime, and the cloud provider exchanges that token for temporary credentials.

    Why OIDC is more secure??
        No long-lived credentials stored, Automatic credential expiration, Access can be restricted per repository, branch, or environment
        Temporary credentials expire in minutes
    Security benefits??
        No static credentials, Short-lived credentials, Fine-grained trust
        No credential rotation, Better auditability

    How long can short-lived credentials stay valid?
        MaxSessionDuration
        Min = 1 HR and Max = 12hr
        We can set this in the Role , inside there in the Maximum session duration

    A GitHub Actions workflow requests an OIDC token from GitHub's OIDC provider. 
    GitHub returns a signed JWT, which is presented to AWS. AWS verifies the JWT's issuer and claims (such as the repository and branch) 
    against its trust policy. If valid, AWS issues temporary credentials that the workflow uses to access AWS resources.

    JWT stands for JSON Web Token, A JWT is a single string with three parts, separated by dots (.)
    xxxxx.yyyyy.zzzzz  Header.Payload.Signature

    Header: The header tells AWS what algorithm(RS256) was used to sign the token
    Payload: It contains, iss, aud, sub, repo, repo_owner, ref, workflow, actor, share, exp

        | Claim        | Meaning                                                 |
        | ------------ | ------------------------------------------------------- |
        | `iss`        | Who issued the token (GitHub's OIDC provider)           |
        | `aud`        | Who the token is intended for (AWS STS in this example) |
        | `sub`        | The subject (often includes the repository and branch)  |
        | `repository` | GitHub repository name                                  |
        | `ref`        | Git branch or tag                                       |
        | `workflow`   | Workflow file name                                      |
        | `actor`      | User or GitHub App that triggered the workflow          |
        | `sha`        | Commit SHA being built                                  |
        | `exp`        | Expiration time of the JWT                              |

    Signature: The signature is generated by GitHub using its private key.
    AWS uses GitHub's public key to verify that:
    the JWT was really issued by GitHub, it hasn't been modified, and it hasn't expired.

    JWT token exp is approx 5min

5) What is an ephemeral runner, and why is it preferred for self-hosted infrastructure?
    
    What is a Runner?
        A runner is simply a machine (VM, physical server, or container) that executes your GitHub Actions workflow.
        
        1. GitHub-hosted Runner
            A GitHub-hosted runner is a machine provided and managed by GitHub.
            You don't create it. You don't patch it. You don't install the operating system.
            
            GitHub automatically: Creates Ubuntu VM, Runs the job, Deletes the VM
            This is why GitHub-hosted runners are already ephemeral.

            Disadvantages:
                Limited customization
                May not access your private network

        2. Self-hosted Runner
            A Self-hosted runner is your own machine. You install the GitHub Runner software on it.
            GitHub doesn't own this machine.

            Why do companies use Self-hosted runners?
                Suppose your deployment needs access to:
                Internal databases
                Private Kubernetes cluster
                Corporate VPN
                Internal Artifactory
                On-premise servers

                GitHub-hosted runners cannot access these private resources.
        
        A GitHub-hosted runner is a virtual machine managed by GitHub that is automatically provisioned 
        for each workflow run and destroyed afterward. It requires no infrastructure management from the user. 
        
        A self-hosted runner, on the other hand, is a machine that you own and manage, 
        such as an EC2 instance, VM, or physical server, on which you install the GitHub Actions 
        runner software. GitHub dispatches workflow jobs to this machine, giving you more 
        control over the environment, network access, installed tools, and hardware.


    What happens after the job finishes?

        Runner stays alive → Persistent
        Runner is destroyed → Ephemeral

        GitHub-hosted runners are always ephemeral.
        Yes, in production environments, companies often use self-hosted runners in an ephemeral fashion. 
        Instead of running long-lived machines, they dynamically provision infrastructure such as EC2 instances.

    What is a GitHub Organization?
        A GitHub Organization is a shared account designed for companies or teams.
        It centrally owns repositories, manages users through roles and teams, and provides 
        organization-wide security, policies, and collaboration features.

    Why not use a personal account?
        Repositories should remain with the company even if employees leave.
        Organizations provide centralized access control, security policies, billing, and collaboration features that personal accounts don't.

    Can an organization own repositories?
        Yes. In enterprise environments, repositories are typically owned by the 
        organization, while developers are granted appropriate access.

    Can one user belong to multiple organizations?
        Yes. A developer or consultant can be a member of multiple
        GitHub Organizations, each with independent permissions.

    Does an organization have owners?
        Yes. Organizations have one or more Owners who can manage organization settings,
        members, billing, security, and repositories.

    Organization

        Organization
        ├── Owners
        ├── Members
        ├── Teams
        ├── Repositories
        └── Outside Collaborators
    
    1. Organization Owners
        Owners are the highest-privileged users in an organization.
        They are like the GitHub administrators for the company.
        Owners control the organization itself.
        
        An owner can:
        Invite users, Remove users, Create teams, Delete teams
        Create repositories, Delete repositories
        Change repository visibility,Configure GitHub Actions
        Manage organization secrets, Configure self-hosted runners
        Configure SSO,Configure billing,View audit logs,Change security policies

    What's the difference between an Owner and a Member?
        Owners administer the organization itself—they manage users, teams, repositories, security settings, runners, billing, and policies. Members typically contribute to repositories

    Why use Teams instead of granting permissions directly?
        Teams simplify access management. Permissions are assigned once to a team, and every member inherits them. This makes onboarding easy.
    
    Can a member access every repository?
        No. Members only see repositories they have been granted access to, either directly or through a team.
    
    What is an Outside Collaborator?
        An Outside Collaborator is someone who is not an organization member but is granted access to one or more specific repositories, commonly used for contractors, auditors, or vendors.

    Can one user belong to multiple teams?
        Yes. For example, a Platform Engineer might belong to both the DevOps team and the Security team, inheriting permissions from both.
    
    How are users managed in a GitHub Enterprise with multiple organizations?
        A user has a single GitHub identity. That identity can be added to one or more organizations within the enterprise. 
        Each organization independently manages the user's organization role, team memberships, and repository permissions. 
        In enterprises using Enterprise Managed Users (EMU), accounts are provisioned centrally through an identity provider 
        like Microsoft Entra ID or Okta using SCIM, rather than being created manually in each organization.


    Scenario 1: New Backend Engineer

    Interviewer: "A backend developer joins today. What steps would you follow?"

    Answer:
    Invite the developer to the GitHub Organization.
    They accept the invitation and become a member.
    Add them to the Backend team.
    The team already has Write access to backend repositories, so the developer automatically receives the correct permissions.
    If required, enforce organization policies such as SSO and 2FA before allowing access.


    Scenario 2: A Contractor Needs Access

    Interviewer: "A contractor needs access to one repository for two weeks. What would you do?"

    Answer:
    I would avoid making them a full organization member if they only need limited access. Instead, I'd add them as an Outside Collaborator with the minimum required repository permissions and remove that access when the engagement ends.
    
    Scenario 3: Developer Moves Teams

    Interviewer: "A developer moves from Frontend to DevOps."

    Answer:
    Remove them from the Frontend team and add them to the DevOps team. Because permissions are team-based, they automatically lose frontend access and gain DevOps access without manually updating each repository.


    Scenario 4: A Team Is Accidentally Deleted

    Interviewer: "What happens if someone deletes the Backend team?"

    Answer:
    All permissions that were granted through that team are removed. Members no longer inherit access from it. This is one reason why only a small number of trusted administrators should have owner privileges.


    Permissions & Access Control
    --------------------------------
        Everything in GitHub Organizations revolves around access control.

        Permission Levels: Think of GitHub permissions as three layers.

        GitHub: Organization Level, Team Level, Repository Level
        Each layer has a different purpose.

        Organization Roles:
            What can this user do within the organization itself?
            There are two default roles you'll use most often.

            Organization -- Owner, Member
            Owners manage the organization.
            Members are regular users.

            Being a Member does not automatically give repository access.
            A member without team membership may have access to zero repositories.

        Repository Roles:
            What can this user do inside this repository?
            Read, Triage, Write, Maintain, Admin

            Read:
                Can: Clone, Pull, View code, Create Issues (if allowed)
                Cannot: Push, Merge, Change settings

            Triage:
                can: Manage Issues, Manage Labels, Manage Discussions, Manage Pull Requests (administratively, such as labeling or closing)
                Cannot: Push code, Modify repository settings

            Write: Most common developer roles
                Can: Clone, Push, Create branches, Create PRs, Merge (subject to branch protection)
            
            Maintain: Maintain is almost like Admin but without destructive capabilities.
                Typical abilities include:
                    Manage labels
                    Manage milestones
                    Configure many repository settings
                    Manage webhooks (depending on repository configuration)
                    Manage collaborators in many cases

                    Cannot:

                    Delete the repository
                    Transfer ownership
                    Change certain administrative/security settings reserved for Admin

                    Typically used for Tech Leads or senior maintainers.

            Admin: Admin has complete repository control.
                Can:
                    Delete repository
                    Change visibility
                    Configure branch protection
                    Manage repository permissions
                    Configure Actions
                    Configure webhooks
                    Configure secrets (repository-level)
                    Manage CODEOWNERS-related settings indirectly through repo administration

                    Usually limited to platform teams or repository owners.

            Interviewer: "A user belongs to three teams. One team has Read access, another has Write access, and the third has Maintain access to the same repository. What is the user's effective permission?"

            Effective Permission = Highest Permission for THAT repository ONLY, Not Highest Permission across the organization.
            The "highest permission wins" rule is evaluated per repository, not across all repositories.

            Priority order:
                Admin, Maintain,Write, Triage, Read

        ***reusable Workflow:(Enterprise level)
        ---------------------
        Of course! So, cross-repo reusable workflows allow you to define a workflow in one repository and then trigger it from another repository. 
        In a personal account, you’d often need a personal access token to authenticate across private repos. 
        But in a GitHub Enterprise organization, you don’t need a personal token—the built-in `GITHUB_TOKEN` is automatically provided for each workflow,

8)What are Labels?
    A label is simply a tag attached to a Pull Request or Issue.
    For example: bug, feature, enhancement
    When you open a PR, someone can attach one or more labels.

    In DevOps and CI/CD, Labels are often used to:
    Trigger or skip specific GitHub Actions workflows.
    Control deployments (for example, only deploy PRs labeled deploy).
    Automatically assign reviewers based on the affected area.
    Prevent merges with labels like do-not-merge or blocked.
    Run extra validation for security or performance changes.

    1. Labels can be added while creating PR and also after creating PR,

             - labeled

        ✅ New commits pushed to a PR that already has the label.

9) Security and Quality

    Security Policy:
        This is simply a document that tells people: If you find a security vulnerability, here's how to report it.

    Dependabot:
        If a project is using libraries which were External and at somepoint a serious vulnerability was discovered
        in that case Dependabot continuously checks your dependencies against GitHub's vulnerability database.
        If your project contains a vulnerable version, it alerts you.
        Example: High Severity
            Jackson 2.13.0  Upgrade to Jackson 2.15.3
            Sometimes Dependabot even creates a PR automatically: You simply review and merge it.
    
    Vulnerabilities: This page shows every dependency with known CVEs.
        Example: Log4j Critical CVE-2021-44228
        You'll see:
        severity, affected package, fixed version, CVE information, upgrade recommendation
        This is extremely useful during audits.Common Vulnerabilities and Exposures

    Malware:
        GitHub also checks whether dependencies are malicious.
        Example: Suppose someone publishes express-secure
        but the package secretly steals AWS credentials.
        GitHub can detect known malicious packages and warn you before they are used.

    Code Scanning:
        This scans your own source code.
        Unlike Dependabot, which checks third-party libraries, Code Scanning looks for problems you introduced.
        Examples:SQL Injection, Hardcoded Passwords, Cross Site Scripting, Command Injection, Unsafe Deserialization, Path Traversal
        
        It can use tools such as: CodeQL, Semgrep, SonarQube (via Actions)Other SAST tools
        Typical workflow:
        Push Code, GitHub Action, Run CodeQL, Upload Results, Security Tab

        You'll see findings like:
        SQL Injection
        Severity: High
        File:
        UserController.java
        Line 52

    Secret Scanning: 
        One of GitHub's best features.

        Suppose you accidentally commit:
            AWS_SECRET_ACCESS_KEY=abc123xyz
        GitHub scans commits and detects:
            AWS keys, Azure keys, GitHub tokens
            Slack tokens, Docker credentials
            Hundreds of other credential formats
        It warns you immediately so you can revoke and rotate the secret.

10)conditional expressions:
    1. if: (Conditional)
        Runs a job or step only if the condition is true.

        if: success()
        if: github.ref == 'refs/heads/main'

    2. needs:
        Specifies dependencies between jobs.

        needs: test
        needs: [test, sonar]

    3. runs-on:
        Specifies which runner executes the job.

        runs-on: ubuntu-latest
        runs-on: windows-latest
        runs-on: self-hosted

    4. uses:
        Uses an action or a reusable workflow.

        uses: actions/checkout@v4

    5. run:
        Runs shell commands.

        run: |
        echo "Hello"
        pwd

    6. with:
        Provides inputs to an action.

        with:
        python-version: "3.12"

    7. env:
        Defines environment variables.

        env:
            APP_ENV: production

    8. name:
        Gives a name to the workflow, job, or step.

        name: CI Pipeline

    9. on:
        Defines which events trigger the workflow.

        on:
        push:
        pull_request:

    10. permissions:
        Specifies the permissions granted to the GITHUB_TOKEN.

        permissions:
        contents: read

    11. strategy:
        Defines a matrix or job strategy.

        strategy:
            matrix:
                python-version: [3.10, 3.11]

    12. timeout-minutes:
        Sets the maximum runtime.

        timeout-minutes: 30

    13. continue-on-error:
        Allows a step or job to fail without failing the workflow.

        continue-on-error: true

    14. defaults:
        Sets default shell or working directory.

        defaults:
        run:
            shell: bash

    15. concurrency:
        Prevents multiple workflow runs from executing simultaneously for the same group.

        concurrency:
            group: production
                cancel-in-progress: true

    16. outputs:
        Exposes values from a job or step to later jobs.

        outputs:
            version: ${{ steps.build.outputs.version }}

    17. services:
        Starts service containers (for example, PostgreSQL or Redis) for a job.

        services:
            postgres:
                image: postgres:15

    18. container:
        Runs the entire job inside a container.

        container:
            image: python:3.12

    | `startsWith()` | Checks if a string starts with a prefix    |

    | `hashFiles()`  | Computes a hash of one or more files       |

12)Jira Integrations
    How does GitHub create a Jira ticket?

    GitHub Actions doesn't create Jira tickets by itself.
    Instead, it calls the Jira REST API.

    The flow is:
    GitHub Actions
        │
        │ HTTPS POST
        ▼
    Jira REST API
        │
        ▼
    Creates Issue

    The workflow sends a JSON payload like:

    {
        "fields": {
            "project": {
            "key": "DEVOPS"
            },
            "summary": "CI Pipeline Failed",
            "description": "Build failed in GitHub Actions",
            "issuetype": {
            "name": "Bug"
            }
        }
    }

    Jira then creates a ticket such as:
    DEVOPS-245
    Summary: CI Pipeline Failed

    Description:
    Repository : payment-service
    Branch     : main
    Workflow   : CI Pipeline
    Failed Job : Unit Tests
    Run URL    : https://github.com/...

    How does GitHub authenticate?
        Just like Slack uses a webhook, Jira uses authentication.
        Usually you store these as GitHub Secrets:
        JIRA_BASE_URL
        JIRA_EMAIL
        JIRA_API_TOKEN
        JIRA_PROJECT_KEY

        Then the workflow uses them to call the Jira API.

    In real companies
        Usually the flow is a little smarter than "create a ticket whenever something fails."
        For example:
        Pipeline Failed
            │
            ▼
        Was this failure already reported?
            │
        ┌── Yes ──► Add a comment/update the existing ticket
        │
        └── No ──► Create a new Jira issue

        Otherwise, if a pipeline fails 100 times, you don't want 100 duplicate Jira tickets.

    Our end goal
    GitHub Actions
        │
        ├── Pipeline Success
        │        │
        │        └── Close existing Jira issue (if one exists)
        │
        └── Pipeline Failure
                │
                ├── Search Jira
                │
                ├── Existing issue?
                │      │
                │      ├── Yes → Add comment & increment failure count
                │      │
                │      └── No → Create new Jira issue
                │
                └── Send Slack notification
