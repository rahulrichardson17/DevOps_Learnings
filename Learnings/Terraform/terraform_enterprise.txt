1) Terraform Open Source vs Terraform Cloud/Enterprise

    Normally you run:
        terraform init
        terraform plan
        terraform apply
    from your laptop or Jenkins server.

    State is usually stored in S3:
        backend "s3" {
            bucket = "terraform-state"
        }

    But in Terraform Cloud (TFC) or Terraform Enterprise (TFE):

        State is stored centrally.
        Terraform runs happen remotely.
        Teams collaborate safely.
        RBAC exists.
        Policy enforcement exists.
        Cost estimation exists.
        VCS integration exists.
        Audit logs exist.

    In simple terms think like
        | Git                | Terraform         |
        | ------------------ | ----------------- |
        | GitHub             | Terraform Cloud   |
        | Local repository   | Local Terraform   |
        | Branch protections | Sentinel policies |
        | Pull requests      | Terraform Runs    |

    Terraform code is executed on Terraform Cloud workers instead of local machines. 
    The plan, policy checks, cost estimation, and apply happen remotely while state 
    is centrally managed.

8) Sentinel

    Sentinel is Terraform's policy-as-code framework.
    It enforces governance.

    Security team says, No public S3 buckets.
    Sentinel blocks the deployment.

    Difference between Sentinel and OPA?

        Sentinel is HashiCorp's proprietary Policy-as-Code framework that 
        integrates tightly with Terraform Cloud/Enterprise.

        OPA (Open Policy Agent) is an open-source, vendor-neutral 
        Policy-as-Code engine that works with Terraform, Kubernetes, APIs, CI/CD pipelines, and many other platforms.

        So,
            Sentinel → Terraform Cloud/Enterprise ecosystem
            OPA → Universal policy engine
        
        Comparison Table
        | Feature            | Sentinel                   | OPA                                           |
        | ------------------ | -------------------------- | --------------------------------------------- |
        | Developed by       | HashiCorp                  | CNCF (Open Source)                            |
        | License            | Proprietary                | Open Source                                   |
        | Language           | Sentinel                   | Rego                                          |
        | Vendor Lock-in     | Yes                        | No                                            |
        | Terraform Support  | Native                     | Via Conftest, Terraform plan evaluation       |
        | Kubernetes Support | No (not primary)           | Excellent (Gatekeeper, admission control)     |
        | CI/CD Integration  | Terraform Cloud            | GitHub Actions, Jenkins, GitLab, ArgoCD, etc. |
        | Cloud Providers    | Mainly HashiCorp ecosystem | AWS, Azure, GCP, Kubernetes, APIs             |
        | Best For           | Terraform Cloud/Enterprise | Multi-platform governance                     |

    
    Sentinel:

        Sentinel is built specifically for the HashiCorp ecosystem.
            It can inspect:
                Terraform plan
                Terraform state
                Terraform configuration
                Workspace variables
            before an apply.

        Example policy: No EC2 instance larger than t3.medium.

        If someone writes:

            resource "aws_instance" "web" {
                instance_type = "m5.large"
            }

            Terraform Cloud executes:
                terraform plan
                    ↓
                Sentinel Policy
                    ↓
                Denied ❌

                Apply never happens.

        Example:
            Only approved regions
            Mandatory tags
            No public S3 bucket
            Production must use encrypted EBS

    OPA (Open Policy Agent):

        OPA is much more generic. It can evaluate almost anything.

        Examples:
            Terraform, Docker, Kubernetes, REST APIs, GitHub Actions, 
            Istio, AWS IAM Policies, JSON/YAML


        OPA policies are written in Rego.

        Example:

            deny[msg] {
                input.resource_type == "aws_s3_bucket"
                input.public == true
                msg := "Public S3 bucket not allowed"
            }

            Terraform plan
                ↓
                OPA
                ↓
                Violation
                ↓
                Pipeline fails


        Terraform + OPA:

            Terraform creates, 
                terraform plan -out=tfplan

            Convert to json
                terraform show -json tfplan

            OPA evaluates:
                plan.json
                ↓
                OPA
                ↓
                Pass/Fail

                If policy fails:
                    CI Pipeline
                    ↓
                    Stop Deployment

                Unlike Sentinel, OPA is not built into Terraform Cloud—you integrate it yourself, often in CI/CD.

        Sentinel and OPA are both Policy-as-Code frameworks, but they target different ecosystems. 
        Sentinel is HashiCorp's proprietary policy engine and integrates natively with Terraform 
        Cloud and Enterprise, allowing organizations to validate Terraform plans before they're applied. 
        
        OPA is an open-source, vendor-neutral policy engine that uses the Rego language and can enforce 
        policies across Terraform, Kubernetes, CI/CD pipelines, APIs, and other systems. If an organization 
        is heavily invested in Terraform Cloud, Sentinel is a natural choice. If it needs centralized policy 
        enforcement across multiple technologies, OPA is generally the better fit.

        Can OPA replace Sentinel?
            Yes, for Terraform policy enforcement OPA can often be used instead of Sentinel, 
            especially in CI/CD pipelines. However, if you're using Terraform Cloud/Enterprise 
            and want native policy checks, policy sets, and run integration, Sentinel provides a 
            smoother experience. Many organizations choose OPA for its flexibility and vendor neutrality, 
            while others use Sentinel because they're already invested in the HashiCorp platform.

17) Workspace variables and variable sets
    A workspace variable belongs to a single Terraform Cloud workspace 
    and is typically used for values specific to that environment, such as ENV or 
    INSTANCE_TYPE. 
    
    A Variable Set is a reusable collection of variables that can be shared across 
    multiple workspaces, making it ideal for common values like cloud credentials, 
    regions, or organization-wide tags. This reduces duplication and simplifies 
    maintenance. If both define the same variable, the workspace variable overrides 
    the value from the Variable Set.

    Think of it like this:
        Variable = Local to one workspace.
        Variable Set = Reusable across many workspaces.

    Sensitive Variables
        Variable Sets also support sensitive values.
        Terraform Cloud hides the value but still injects it during runs.

    Can Workspace Variables Override Variable Set Values?
        Yes. Suppose your Variable Set contains
        INSTANCE_TYPE = t3.micro, But your production workspace defines
        INSTANCE_TYPE = m6i.large
        The workspace-specific variable takes precedence over the Variable Set.
        This lets you define common defaults while allowing exceptions.

    When you mark it as Sensitive:
        Terraform Cloud encrypts and stores it securely.
        The value is hidden in the UI.
        The value is not shown in logs or plans (unless your Terraform code explicitly outputs it, which should be avoided).
        Terraform still injects it into the run.

18) VCS Integration

    How does Terraform Cloud integrate with GitHub?
        VCS Integration

    Terraform Cloud monitors a Git repository. Whenever a commit or pull request 
    is pushed, it automatically queues a run, performs a plan, evaluates Sentinel policies, 
    estimates costs, and optionally applies the changes after approval.

    VCS = Version Control System
        Examples:
            GitHub
            GitLab
            Bitbucket
            Azure DevOps

    VCS Integration means:
        Terraform Cloud connects directly to your Git repository and 
        automatically runs Terraform whenever your infrastructure code changes.
    
    Without VCS:
        Everyone needs Terraform installed.
        Everyone needs AWS credentials.
        No central history.
        No approvals.
        No audit trail.
        Easy to make mistakes.

    GitHub sends a webhook to Terraform Cloud.Terraform Cloud immediately starts a new run.

    After the PR is merged into main, Terraform Cloud performs the real apply 
    (depending on the workspace configuration).

    Does Terraform Cloud store my Terraform code?
        Terraform Cloud does not copy and permanently store your repository 
        as the source of truth. Your Git repository remains the source of truth. 
        During a run, Terraform Cloud temporarily clones the configured repository 
        (or working directory), executes Terraform, and records the run details 
        and state. Your code continues to live in GitHub, GitLab, Azure DevOps, etc.
    
    Github credentials
        When you click Connect GitHub in Terraform Cloud:GitHub redirects you to its login page (if you're not already signed in).
        You log in to GitHub directly.Do you want to authorize Terraform Cloud to access your repositories
        You click Authorize Terraform Cloud
        
        GitHub issues an OAuth token to Terraform Cloud.
        Terraform Cloud stores that authorization securely—not your GitHub password.

22)Private Module Registry
    The Private Module Registry is a central repository inside a Terraform 
    Cloud Organization where you can store and share your own reusable Terraform 
    modules.

    Think of it like:
        Terraform Registry → Public modules shared by the community.
        Private Module Registry → Your company's internal modules.

    Why do we need it?
        Suppose every team needs to create a VPC.
        Without a Private Registry, each team writes its own VPC module:

        Team A
            vpc.tf

        Team B
            vpc.tf

        Team C
            vpc.tf

        Problems:
            Duplicate code
            Different implementations
            Hard to maintain
            Inconsistent standards

        Every team uses the same approved modules.

    Where are the modules stored?
        A common misconception is that Terraform Cloud stores the module code itself.
        Typically, the source code remains in a Git repository (for example, GitHub, 
        GitLab, or Azure DevOps). The Private Module Registry indexes and publishes 
        that repository as a versioned Terraform module.

        Terraform Cloud uses Git tags as module versions.
        
        In Terraform Cloud:
            Organization
            ↓
            Registry
            ↓
            Publish Module

        Choose:
            GitHub repository
            Module name
            Provider (AWS, Azure, GCP, etc.)

        Terraform Cloud imports the module. The module appears in the Private Registry.
        Now anyone in the organization can use it.

23) Module Sharing

    Module Sharing is the ability to reuse the same Terraform module across 
    multiple workspaces, projects, or teams instead of copying the code.

    Private Module Registry = Where modules are published and stored (catalog).
    Module Sharing = How those published modules are made available and reused by other teams/workspaces.

    Is Module Sharing a separate feature from the Private Module Registry?
        They're closely related. The Private Module Registry is where reusable 
        modules are published and versioned. Module Sharing refers to making 
        those published modules available for multiple workspaces 
        and teams to consume, so infrastructure code is standardized and 
        reused instead of duplicated.

24)Terraform Agents
    
    Normally, Terraform Cloud executes your runs on HashiCorp-managed workers.
    
        Developer
            │
            ▼
        Terraform Cloud
            │
            ▼
        HashiCorp Managed Runner
            │

        AWS / Azure / GCP

    This works well if your infrastructure is publicly accessible.

    Problem

        Suppose your company has:

        Private AWS VPC
        Private Kubernetes cluster
        On-premises VMware
        Private databases
        Internal APIs

        These resources are not reachable from the public internet.
        HashiCorp's runners cannot access them.

    Terraform Agent:
        You install a Terraform Agent inside your private network.
        Developer
            │
            ▼
        Terraform Cloud
            │
            ▼
        Terraform Agent
        (Inside your VPC)
            │
            ▼
        Private AWS Resources

        The agent receives jobs from Terraform Cloud and executes them locally within your network.

    Terraform Cloud vs Terraform Agent
        | Managed Execution                   | Terraform Agent                    |
        | ----------------------------------- | ---------------------------------- |
        | Runs on HashiCorp infrastructure    | Runs inside your infrastructure    |
        | Best for public cloud resources     | Best for private networks          |
        | No infrastructure to manage         | You install and maintain the agent |
        | Cannot reach private-only endpoints | Can reach private resources        |

    Terraform Agents allow Terraform Cloud to execute runs inside an 
    organization's own network instead of on HashiCorp-managed infrastructure. 
    They're used when Terraform needs access to private resources, such as private 
    VPCs, private Kubernetes clusters, on-premises environments, or internal APIs. 
    The agent securely polls Terraform Cloud for work, executes the Terraform run 
    locally, and reports the results back to Terraform Cloud, while Terraform Cloud 
    continues to provide state management, run history, policies, and governance.

    Terraform Agent allows Terraform Cloud to execute Terraform runs inside 
    our local/private network instead of on HashiCorp-managed infrastructure.

    Execution Modes:
        Three execution modes.
        
        Remote
            Terraform Cloud executes.
        Local
            Your laptop executes.
            Terraform Cloud stores state.
        Agent
            Terraform Agent executes.

26) Different ways of running Terraform in Terraform Cloud

        Instead of a developer pushing code or using the CLI, 
        another application calls the Terraform Cloud API to start a run.

        A REST API request starts the Terraform run.
        GitHub Actions, Azure DevOps can all trigger Terraform Cloud via API.

        API-driven runs allow external applications or CI/CD pipelines to start Terraform runs by calling the Terraform Cloud REST API.

    2. CLI-Driven Workflow
        Normally with Terraform Cloud, code comes from GitHub.
        But what if you're developing locally?

        Terraform Cloud supports a CLI-driven workflow.

        You execute:
            terraform plan
            terraform apply

        from your laptop. However,the execution happens in Terraform Cloud.

        CLI-driven workflow lets developers use normal Terraform CLI commands while Terraform Cloud performs the execution remotely and stores the state.

29) Merge 

    By default, Terraform Cloud does not automatically apply changes after 
    a merge unless the workspace is configured for Auto Apply.

    Terraform Cloud already knows the Terraform workflow. You do NOT write a 
    pipeline with terraform init, plan, apply, or Sentinel policy checks. 
    Terraform Cloud orchestrates all of that automatically.

    Does Terraform Cloud know when to run Sentinel?
        Yes. Suppose your workspace has a Policy Set attached.

    If you don't connect GitHub to Terraform Cloud and you don't have GitHub Actions, what happens?"
        Nothing executes. GitHub only stores the code and pull request. 
        Terraform doesn't run automatically because there's no system watching 
        the repository. To execute Terraform, you need a runner such as Terraform 
        Cloud with VCS integration, GitHub Actions, Jenkins, Azure DevOps, or 
        another CI/CD tool. Without one of these, pushing code or creating a PR 
        does not trigger any Terraform operations.

    Can Terraform Cloud automatically run terraform fmt and terraform validate before I raise the PR?
        No. Terraform Cloud does not automatically run formatting or 
        linting checks on every branch push.

) Questions

        Why Terraform Cloud instead of S3 backend?
            Terraform Cloud provides remote execution, RBAC, audit logs, 
            policy-as-code, cost estimation, centralized state management, 
            VCS integration, and approval workflows. S3 only provides remote state storage.

        How do you manage environments?
            Separate workspaces are used for dev, qa, and prod with separate state files, 
            variables, permissions, and approval workflows.

        How do you enforce governance?
            Sentinel policy sets are attached to workspaces to enforce security, 
            compliance, tagging, approved regions, encryption, and cost controls.

        How is state locking handled?
            Terraform Cloud automatically locks state during runs to prevent 
            concurrent modifications.

        What happens when a Sentinel policy fails?
            The run stops after planning, the policy check fails, 
            and the apply phase is blocked until the violation is fixed.

        Can Terraform Cloud run Terraform locally?
            Yes.
            Execution modes:
                Remote
                Local
                Agent

        Do we need GitHub Actions to trigger Terraform Cloud?
            No. Terraform Cloud's VCS integration is sufficient for the standard 
            workflow. It uses GitHub webhooks to automatically trigger speculative 
            plans on pull requests and normal runs after code is merged into the 
            tracked branch. GitHub Actions are optional and are commonly used for 
            additional validation steps such as terraform fmt, terraform validate, 
            TFLint, Checkov, or other security and quality checks before Terraform 
            Cloud executes the infrastructure changes.

        terraform fmt (Terraform Format)
            What it does?
                terraform fmt automatically formats Terraform code into the 
                standard Terraform style.

                terraform fmt -check (this only checks, it will not format)

        terraform validate

        Terraform Cloud OIDC Flow
            Terraform Cloud gets a temporary AWS credential.
            Provider Type:
            OpenID Connect

            Provider URL:
            https://app.terraform.io

            Audience:
            aws.workload.identity


            create IAM role
                {
                    "Version": "2012-10-17",
                    "Statement": [
                        {
                        "Effect": "Allow",
                        "Principal": {
                            "Federated": "arn:aws:iam::<ACCOUNT_ID>:oidc-provider/app.terraform.io"
                        },
                        "Action": "sts:AssumeRoleWithWebIdentity",
                        "Condition": {
                            "StringEquals": {
                            "app.terraform.io:aud": "aws.workload.identity"
                            }
                        }
                        }
                    ]
                }

                Enable OIDC authentication
                TFC_AWS_PROVIDER_AUTH = true

                Mark as:

                Environment Variable
                Provide IAM Role ARN
                    TFC_AWS_ROLE_ARN = arn:aws:iam::<ACCOUNT_ID>:role/terraform-cloud-deploy-role

                Also as:
